# Ghostcam — Authentication and Certificate Lifecycle

**Status:** Draft

---

## 1. Overview

This document specifies the mutual TLS certificate model used to authenticate cameras to the server, the enrollment and unregistration flows, and the certificate revocation mechanism.

All camera-to-server QUIC connections use mutual TLS. The server presents its server certificate; cameras present two client certificates. Both sides verify before any application data is exchanged. This gives the server cryptographic proof of device identity and group membership before the ingest slot is created.

Certificate storage on the camera is specified in `camera-firmware.md`. Server-side connection verification is specified in `ingest.md`.

---

## 2. Two-Layer Certificate Model

Cameras carry two certificates with distinct purposes and independent lifecycles.

### 2.1 Device identity certificate

| Property | Value |
|----------|-------|
| Issued by | Ghostcam CA at manufacturing or flash time |
| Subject | Hardware identifier (SoC serial number in CN or SAN) |
| Validity | Permanent — never rotates |
| Purpose | Proves the device is genuine Ghostcam hardware |
| Scope | Global — not tied to any user account |

The device identity certificate allows the server to refuse connections from non-Ghostcam devices before any user account is involved. It is necessary but not sufficient for the server to admit a connection — it proves the device is legitimate hardware, not that it belongs to any enrolled user account.

### 2.2 User association certificate

| Property | Value |
|----------|-------|
| Issued by | Server during enrollment |
| Subject | `device_id` of the camera |
| Validity | Permanent — revocation only |
| Purpose | Gates access to an enrolled user account's ingest slots and egress handles |
| Scope | Scoped to a single `device_id` |

The user association certificate is issued after the user associates the device with their account. It is revocable independently of the device identity certificate, allowing a user to remove a device from their account without affecting the device's ability to be re-enrolled by another user.

---

## 3. Server Certificate Verification

On each incoming QUIC connection the server performs the following checks at the TLS layer before any application data is processed:

1. Verify the device identity certificate against the Ghostcam CA root
2. Confirm the device is known hardware (CN/SAN matches a registered device identifier in the Ghostcam CA)
3. Verify the user association certificate against the Ghostcam CA
4. Confirm the user association certificate serial number is not on the server-internal revocation list (see §6)

If any check fails the connection is rejected at the TLS layer. A camera in registration mode presents only its device identity certificate — the server recognises the absence of a user association certificate and routes the connection to the enrollment handler rather than rejecting it outright (see §4).

## 4. Enrollment Flow

Enrollment associates a device with a user account and issues a user association certificate. It uses a QR code system to ensure enrollment is proximity-gated and trustless — no out-of-band channel or polling is required.

### 4.1 Registration mode

On startup, if the camera has no valid user association certificate, it enters registration mode. In registration mode the camera does not attempt a normal QUIC connection. Instead it waits to receive an enrollment token delivered to the camera by the user pointing their phone at a QR code displayed in the client app. The token is short-lived and generated by the server, proving the server authorised this specific enrollment and binding it to the initiating user account.

### 4.2 Flow

```
User                    Client App              Server                  Camera
 |                          |                     |                       |
 |-- initiates enrollment ->|                     |                       |
 |   (provides device ID)   |                     |                       |
 |                          |-- POST /cameras/enroll ->|                  |
 |                          |   { device_id }     |                       |
 |                          |                     |-- generate token      |
 |                          |                     |-- encode as QR        |
 |                          |<-- QR code ---------|                       |
 |-- scans QR code -------->|                     |                       |
 |                          |-- deliver token ---------------------------------------->|
 |                          |                     |                       |
 |                          |                     |<-- QUIC/mTLS connect -|
 |                          |                     |    (device cert +     |
 |                          |                     |     token)            |
 |                          |                     |-- verify device cert  |
 |                          |                     |-- verify token        |
 |                          |                     |-- confirm unenrolled  |
 |                          |                     |-- open Commands stream|
 |                          |                     |<-- csr alert ---------| (camera generates key pair)
 |                          |                     |-- sign CSR            |
 |                          |                     |-- cert_refresh cmd -->|
 |                          |                     |                       |-- store cert + key
 |                          |                     |<-- ack ---------------|
 |                          |                     |-- mark enrolled in DB |
 |                          |                     |-- close connection    |
 |                          |<-- 200 OK -----------|                       |
```

### 4.3 Enrollment JWT

The QR code encodes a signed JWT. The JWT is signed by the server using its CA key (ES256 — ECDSA with P-256 and SHA-256). See `pki.md` §8 for the signing model.

```json
{
  "iss": "ghostcam-server",
  "exp": 1234567890,
  "jti": "<uuid>",
  "server_addr": "192.168.1.10:4433",
  "display_name": "Back Door",
  "wifi": [
    { "ssid": "HomeNetwork", "psk": "password123" }
  ]
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `iss` | yes | `"ghostcam-server"` |
| `exp` | yes | Expiry timestamp (Unix seconds). JWT is valid for 10 minutes from generation. |
| `jti` | yes | Unique token ID — UUID stored in `enrollment_tokens` table for replay prevention |
| `server_addr` | yes | Server address the camera should connect to for enrollment |
| `display_name` | no | Initial display name for the camera in the UI |
| `wifi` | no | Array of WiFi credentials to store on the camera |

The camera does not verify the JWT signature at scan time — verification happens server-side. The camera checks only the `exp` field (clock skew tolerance: ±5 minutes). See §8 for the known limitation and planned hardening.

### 4.4 Steps

1. User initiates enrollment in the client app, optionally entering a display name
2. Client app calls `POST /api/v1/cameras/enroll` with optional `display_name`
3. Server generates a JWT with a `jti` UUID, records the `jti` in `enrollment_tokens`, encodes the JWT as a QR code, and returns it to the client app. The QR is valid for 10 minutes.
4. User holds their phone screen displaying the QR code up to the camera lens
5. Camera decodes the QR, checks `exp`, stores any WiFi credentials to `/var/ghostcam/networks.conf`, stores `server_addr` to `/etc/ghostcam/server.addr`
6. Camera initiates an enrollment QUIC connection to the decoded `server_addr`, presenting only its device identity certificate
7. Server verifies the device cert, receives the JWT on the enrollment connection, verifies the JWT signature, checks `exp`, checks `jti` against `enrollment_tokens` — rejects if already claimed
8. Server marks the `jti` as claimed in `enrollment_tokens`
9. Server creates a `cameras` row with a new UUID `device_id`, the device cert fingerprint, and the `display_name` from the JWT (or "New Camera" if absent)
10. Server opens a Commands stream toward the camera
11. Camera generates a P-256 key pair locally, opens an Alerts stream, sends a `csr` alert
12. Server signs the CSR — subject `CN={device_id}` — and delivers the certificate via `cert_refresh`, which also carries the server's CA cert
13. Camera stores the user association cert to `/etc/ghostcam/user.crt`, the key to `/etc/ghostcam/user.key`, and the CA cert to `/etc/ghostcam/ca.crt`
14. On `server-solo`: camera performs TOFU — stores the server TLS fingerprint to `/etc/ghostcam/server.pin`
15. Camera sends `ack`
16. Server marks the camera as enrolled only after receiving the `ack`
17. Server closes the enrollment QUIC connection
18. Camera exits registration mode and proceeds with the normal startup sequence

Step 16 is the critical ordering constraint. If the server marks the camera enrolled before the `ack`, a crash between steps 13 and 15 would leave the camera unable to connect (cert stored but ack not sent) while the server considers it enrolled and would reject re-enrollment.

### 4.5 `cert_refresh` payload during enrollment

During enrollment the `cert_refresh` command carries two fields:

```json
{
  "type": "cert_refresh",
  "seq": 7,
  "cert_pem": "<PEM-encoded user association certificate>",
  "ca_pem": "<PEM-encoded server CA certificate>"
}
```

The `ca_pem` field is only present during the initial enrollment. Subsequent `cert_refresh` commands (if certificate rotation is implemented in future) carry only `cert_pem`.

### 4.6 Enrollment connection lifecycle

The enrollment QUIC connection is a lightweight variant of the normal connection. No `IngestSlot` is created. Only two streams are opened: an Alerts stream (camera → server) for the `csr` alert and `ack`, and a Commands stream (server → camera) for the `cert_refresh` command. No Video, Audio, or telemetry streams are opened. The connection is closed by the server after the `ack` is received.

## 5. Unregistration

Unregistration removes a device from a user account and returns it to device-identity-only state, ready for re-enrollment.

### 5.1 Flow

1. Server sends `unregister` command to camera via Commands stream
2. Camera sends `ack` on Alerts stream
3. Camera clears all enrollment state from local storage:
   - `/etc/ghostcam/user.crt` and `user.key`
   - `/etc/ghostcam/ca.crt`
   - `/etc/ghostcam/server.pin` (if present)
   - `/etc/ghostcam/server.addr`
4. Camera closes the QUIC connection and enters registration mode
5. Server hard-deletes the `cameras` row — all associated data (telemetry history in Redis, segment metadata in Redis, revocation list entry) is purged
6. Server adds the user association certificate serial number to the revocation list in Redis

The server MUST NOT add the certificate to the revocation list before receiving the `ack`. Revoking before the camera has cleared its local copy would leave the camera holding a revoked cert, preventing reconnection and requiring manual intervention to recover.

The device identity certificate is unaffected by unregistration. The camera is immediately available for re-enrollment by the same or a different user.

### 5.2 Offline unregistration

If the camera is offline when unregistration is requested, the server queues the `unregister` command and delivers it on the next connection. The user association certificate is placed in a pending-revocation state — the server continues to accept connections using this cert until the command is delivered and acknowledged, at which point the serial number is added to the revocation list.

## 6. Certificate Revocation

The server maintains an internal revocation list of certificate serial numbers stored in Redis. This list is checked at connection time as part of the mTLS verification step (see §3, step 4).

### 6.1 Revocation list structure

Revoked serial numbers are stored in Redis as a set:

```
revoked_certs  ->  Set<serial_number>
```

Each entry is the hex-encoded serial number of a revoked user association certificate. Device identity certificates are never revoked — they are permanent and their validity is tied to hardware authenticity.

### 6.2 Revocation at connection time

On each incoming QUIC connection the server checks the user association certificate's serial number against the Redis revocation set before creating an IngestSlot. If the serial number is present, the connection is rejected at the TLS layer.

### 6.3 Revocation TTL

User association certificates do not expire — they are permanent. Revocation list entries therefore have no natural expiry and are retained indefinitely unless manually cleaned up after a device is destroyed or permanently decommissioned.

### 6.4 Multi-server deployments

In a multi-server deployment, the Redis revocation set is shared across all server nodes via the Redis cluster. Revocation propagates immediately to all nodes that share the same Redis cluster. Full multi-server topology is deferred to `multi-server.md`.

## 7. Ghostcam CA

The Ghostcam CA is the root of trust for all device identity and user association certificates. It is operated by Ghostcam and is not exposed to end users.

| Certificate type | Issuer | Lifetime |
|-----------------|--------|----------|
| Device identity | Ghostcam CA (offline root) | Permanent |
| User association | Ghostcam CA (online intermediate) | Permanent — revocation only |
| Server certificate | Ghostcam CA (online intermediate) | Standard TLS rotation |

Device identity certificates are issued by an offline root CA at manufacturing time to minimise exposure of the root key. User association and server certificates are issued by an online intermediate CA operated by the server infrastructure.

## 8. Open Questions

| Question | Notes |
|----------|-------|
| Enrollment token encoding — camera-side JWT verification | The camera does not verify the JWT signature at scan time (see §4.3). A physically-present attacker with a forged QR could cause the camera to store attacker-controlled WiFi credentials before the server rejects the invalid `jti`. Camera-side verification (embedding the CA cert in the QR payload) is a planned hardening measure. See `pki.md` §8. |
| Intermediate CA rotation | The procedure for rotating a compromised or expired `server-multi` intermediate CA is outlined in `pki.md` §4.3 but not fully implemented. Cameras enrolled under the old intermediate need continued support during the rotation window. |
