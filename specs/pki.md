# Ghostcam — PKI and Certificate Model

**Status:** Draft

---

## 1. Overview

This document specifies the public key infrastructure for Ghostcam: the certificate hierarchy, how trust is established between cameras and servers, the CA operational model for both server variants, and the enrollment JWT signing model.

Certificate storage on the camera is specified in `camera-firmware.md` §9. The enrollment flow that uses these certificates is specified in `auth.md`. Server-side connection verification is specified in `ingest.md` §2.

---

## 2. Certificate Hierarchy

Ghostcam uses a two-tier CA hierarchy for `server-multi` and a self-signed single-tier CA for `server-solo`.

### 2.1 `server-multi` hierarchy

```
Ghostcam Root CA  (offline, air-gapped)
    └── Ghostcam Intermediate CA  (online, shared across all server-multi instances)
            ├── User association cert  (per enrolled camera)
            ├── User association cert
            └── ...
```

The root CA is kept offline and is used only to sign the intermediate CA cert. It never touches production infrastructure. The intermediate CA cert and key are stored in a secrets management service accessible to all `server-multi` instances (see §4.2).

### 2.2 `server-solo` hierarchy

```
Instance CA  (self-signed, generated on first startup)
    ├── User association cert  (per enrolled camera)
    ├── User association cert
    └── ...
```

The instance CA acts as both root and intermediate. It is generated once on first startup, stored on the server's data volume, and reused on subsequent startups. There is no Ghostcam root CA involvement for `server-solo`.

### 2.3 Key algorithm

All CA keys and camera-generated keys use **P-256 (ECDSA)**. P-256 produces significantly smaller certificates than RSA-2048 (approximately 250 bytes DER vs 900 bytes), which matters for the QR code enrollment payload and for QUIC handshake overhead. P-256 is supported by rustls and is FIPS 140-3 compliant via aws-lc-rs.

## 3. Certificate Types

There are three certificate types in the Ghostcam system. Device identity certificates, which previously existed in the spec, have been removed — cameras generate their own self-signed device certs which serve as transport credentials only (see §5).

### 3.1 Ghostcam Intermediate CA certificate (`server-multi` only)

| Property | Value |
|----------|-------|
| Issuer | Ghostcam Root CA |
| Subject | `CN=Ghostcam Intermediate CA` |
| Key usage | `keyCertSign`, `cRLSign` |
| Validity | 5 years |
| Stored | Secrets manager (e.g. AWS Secrets Manager, Fly.io secrets) |
| Purpose | Signs user association certs during enrollment; signs enrollment JWTs |

### 3.2 Instance CA certificate (`server-solo` only)

| Property | Value |
|----------|-------|
| Issuer | Self-signed |
| Subject | `CN=Ghostcam Instance CA` |
| Key usage | `keyCertSign`, `cRLSign` |
| Validity | 20 years |
| Stored | `/var/ghostcam/ca.crt` + `/var/ghostcam/ca.key` |
| Purpose | Signs user association certs during enrollment; signs enrollment JWTs |

### 3.3 User association certificate

| Property | Value |
|----------|-------|
| Issuer | Ghostcam Intermediate CA (`server-multi`) or Instance CA (`server-solo`) |
| Subject | `CN={device_id}` where `device_id` is the server-assigned UUID |
| Key usage | `digitalSignature`, `clientAuth` |
| Validity | Permanent — revocation only (see `auth.md` §6) |
| Generated by | Camera (key pair generated locally, never leaves device) |
| Stored on camera | `/etc/ghostcam/user.crt` + `/etc/ghostcam/user.key` |
| Purpose | mTLS client certificate presented on every QUIC connection post-enrollment |

The user association certificate's subject CN carries the server-assigned `device_id` UUID. The server extracts this on each connection to identify the camera without a database lookup at the TLS layer.

## 4. CA Operations

### 4.1 `server-solo` — instance CA bootstrap

On first startup, if `/var/ghostcam/ca.crt` does not exist:

1. Generate a P-256 key pair
2. Issue a self-signed CA certificate with a 20-year validity and `keyCertSign` key usage
3. Write cert to `/var/ghostcam/ca.crt` and key to `/var/ghostcam/ca.key`
4. Print a warning to stdout:

```
IMPORTANT: Back up /var/ghostcam/ca.key
Losing this file requires re-enrolling all cameras.
```

On subsequent startups the existing CA cert and key are loaded from disk. They are never regenerated unless manually deleted.

### 4.2 `server-multi` — intermediate CA storage

The Ghostcam Intermediate CA cert and key are stored in the deployment's secrets management service and injected into server instances as environment variables or mounted secrets at startup:

```
GHOSTCAM_CA_CERT=<PEM>
GHOSTCAM_CA_KEY=<PEM>
```

The server loads these at startup and holds them in memory for the lifetime of the process. They are never written to disk on server instances.

### 4.3 Ghostcam Root CA operations

The root CA is operated offline. Its only function is to sign the Ghostcam Intermediate CA certificate. The root CA key is stored on an air-gapped machine or hardware security module and is never connected to the internet.

To issue a new intermediate CA certificate (e.g. on intermediate rotation or compromise):

1. Generate a new P-256 key pair for the intermediate
2. Create a CSR for the new intermediate
3. Sign the CSR with the root CA key (offline operation)
4. Deploy the new intermediate cert and key to secrets management
5. Roll out to all `server-multi` instances

Cameras enrolled before the intermediate rotation continue to work — their user association certs were signed by the previous intermediate. The server must verify user association certs against both the current and any recently retired intermediate during a rotation window. This is a planned future operational procedure; intermediate rotation is not implemented in v1.

### 4.4 CSR signing during enrollment

When a camera sends a `csr` alert during enrollment, the server:

1. Parses the PEM-encoded CSR
2. Verifies the CSR signature (proves the camera holds the corresponding private key)
3. Extracts the public key from the CSR
4. Issues a user association certificate:
   - Subject: `CN={device_id}` (server-assigned UUID for this enrollment)
   - Issuer: Instance CA (`server-solo`) or Ghostcam Intermediate CA (`server-multi`)
   - Validity: No expiry (permanent)
   - Key usage: `digitalSignature`, `clientAuth`
5. Returns the signed certificate via `cert_refresh` command

The private key generated by the camera never leaves the camera. The server signs only the public key from the CSR.

## 5. Camera Device Certificate

Each camera generates a self-signed device certificate on first startup. This certificate is a transport credential only — it does not carry application-level identity and is not issued or verified by any CA.

| Property | Value |
|----------|-------|
| Issuer | Self-signed |
| Subject | `CN=ghostcam-device` (static — identity is not carried here) |
| Key usage | `digitalSignature`, `clientAuth` |
| Validity | 20 years |
| Generated | On first boot if `/etc/ghostcam/device.crt` does not exist |
| Stored | `/etc/ghostcam/device.crt` + `/etc/ghostcam/device.key` |
| Purpose | mTLS client certificate; public key fingerprint used as stable hardware identifier |

The server derives the camera's stable hardware identity from the SHA-256 fingerprint of the device certificate's public key. This fingerprint is stored in `cameras.cert_fingerprint` in the application database and is used to look up the camera's enrollment record on each QUIC connection.

The device certificate is never rotated. If a camera is re-enrolled after unregistration, the same device certificate is presented and the server creates a new `cameras` row with the same `cert_fingerprint` and a new `device_id` UUID — the server intentionally has no memory of the previous enrollment.

## 6. Server TLS Certificate

### 6.1 `server-solo`

The server generates a self-signed TLS certificate on first startup if `/var/ghostcam/server.crt` does not exist:

1. Generate a P-256 key pair
2. Issue a self-signed certificate with `serverAuth` key usage and a 20-year validity
3. Write cert to `/var/ghostcam/server.crt` and key to `/var/ghostcam/server.key`

This certificate is used for both QUIC (camera connections) and HTTP (browser UI). The browser UI is served over plain HTTP on the LAN — the TLS cert is used only for QUIC.

Cameras verify this certificate via TOFU (trust on first use) during enrollment. On the enrollment QUIC connection the camera accepts the server's certificate without verification and pins its fingerprint to `/etc/ghostcam/server.pin`. On all subsequent QUIC connections the camera verifies the server certificate against the pinned fingerprint.

### 6.2 `server-multi`

The server uses a standard CA-signed TLS certificate issued via ACME (Let's Encrypt) for the public domain. This certificate is managed by the deployment infrastructure (Fly.io handles ACME automatically for configured domains) and rotated before expiry without server involvement.

Cameras verify this certificate using the system CA bundle — standard TLS verification. No pinning is needed for `server-multi` because the certificate is issued by a trusted public CA.

The same certificate is used for both QUIC (camera connections) and HTTPS (browser UI and API).

## 7. Camera Trust Store

After enrollment, the camera holds the following trust material:

| File | Content | Used to verify |
|------|---------|---------------|
| `/etc/ghostcam/ca.crt` | Instance CA cert (`server-solo`) or Ghostcam Intermediate CA cert (`server-multi`) | User association certificate on each QUIC connection |
| `/etc/ghostcam/server.pin` | SHA-256 fingerprint of server TLS cert (`server-solo` only) | Server TLS certificate on each QUIC connection |

For `server-multi` connections, the camera uses the system CA bundle to verify the server's TLS cert — no `server.pin` file is needed.

On unregistration, both files are deleted. The camera returns to an unverified state and must re-enroll to establish a new trust relationship.

## 8. Enrollment JWT Signing

Enrollment JWTs (see `auth.md` §4) are signed by the server using its CA key — the Instance CA key (`server-solo`) or the Ghostcam Intermediate CA key (`server-multi`). The JWT algorithm is **ES256** (ECDSA with P-256 and SHA-256), consistent with the P-256 key algorithm used throughout.

The camera does not verify the JWT signature at scan time — verification happens server-side when the camera presents the JWT on its enrollment QUIC connection. See `auth.md` §8 for the known limitation and planned hardening.

The `jti` claim in the JWT corresponds to the primary key in the `enrollment_tokens` table, used for replay prevention.

---

## 9. Revocation

Certificate revocation is specified in `auth.md` §6. In summary: user association certificate serial numbers are stored in a Redis set (`revoked_certs`) and checked on every incoming QUIC connection. The revocation list is cached in memory and refreshed every 60 seconds. Device certificates are never revoked.

There is no CRL or OCSP infrastructure. Revocation is server-internal only.

## 10. Open Questions

| Question | Notes |
|----------|-------|
| Intermediate CA rotation procedure | The procedure for rotating a compromised or expired intermediate CA cert is outlined in §4.3 but not fully specified. Cameras enrolled under the old intermediate need to continue working during the rotation window. A dual-trust approach (accept both old and new intermediate during transition) is the likely solution. |
| Root CA hardware | The offline root CA should ideally be stored on a hardware security module (HSM) or a dedicated air-gapped machine. The specific hardware and operational procedure are not specified. |
| `server.pin` rotation on `server-solo` | If the operator regenerates the server TLS cert (e.g. after a key compromise), all enrolled cameras have stale `server.pin` entries and will refuse to connect. A recovery procedure — either re-enrollment or a server-side mechanism to push a new pin — is needed but not yet specified. |
